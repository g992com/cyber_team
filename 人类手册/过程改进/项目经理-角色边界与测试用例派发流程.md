# 项目经理角色边界与测试用例派发流程

本文档总结「项目经理代写测试用例」越界行为的原因分析，以及当用户需要功能测试用例时应遵循的正确流程。供人类与 Agent 在项目经理会话中参考，避免同类越界。

---

## 1. 背景与问题

用户提出：「根据这份需求，我需要最终得到一份功能测试用例，开始你的工作吧。」

若项目经理在该会话中**亲自编写**功能测试用例文档并交付，即构成**越界**：测试计划/用例的正式产出归属 **qa（测试工程师）**，项目经理不得代为撰写。

---

## 2. 越界原因分析

### 2.1 规范已有明确约定

- **roles/roles.yaml**：测试工程师（qa）负责「测试策略、用例、执行、质量门禁」，phase_ids 含 testing。
- **mapping/phase-role-skill.yaml**：testing 阶段产出由 qa 负责。
- **项目经理职责**：派发任务、回收各角色产出、据此推进阶段；**不撰写**需求/设计/代码/测试的评审结论或评审记录，也不代写各阶段的正式交付物（含测试用例）。

### 2.2 实际发生偏差的原因

1. **对用户意图的曲解**  
   将「开始你的工作吧」理解为「在当前会话内把可交付物做出来」，而不是「按规范推进：派活给对应角色并回收产出」。

2. **默认代做而非默认派发**  
   在未确认「是否有测试工程师会话」的情况下，选择了「由本会话代做并注明越界」，而不是优先「只派发任务 + 提示用户创建/使用 qa 会话」。

3. **例外条款被误用**  
   「若用户**坚持**在本会话内完成，可注明越界」的前提是用户**明确坚持**不另开角色会话。用户并未如此要求，不应主动走该例外。

### 2.3 小结

越界的本质是：**在应仅做「派发与回收」的场景下，代替负责角色产出了正式交付物**。纠正要点：严格区分「项目经理该做的」与「测试工程师该做的」，默认只做前者。

---

## 3. 正确流程（用户需要功能测试用例时）

当用户表达「需要功能测试用例」或「根据需求产出测试用例」时，项目经理应按下述流程执行，**不亲自编写测试用例**。

### 3.1 澄清目标与归属

- 确认用户需要的是「功能测试用例」文档，且该产出的**正式归属角色**为 **qa（测试工程师）**。
- 若有需求文档、原型或截图，作为任务卡的**输入材料**写明，供 qa 会话使用。

### 3.2 仅执行项目经理动作

1. **编写任务卡**  
   包含：任务类型（功能测试用例编写）、当前阶段（testing）、目标、输入材料（需求文档/链接、截图、约束）、输出要求（用例文档格式、覆盖范围）、验收标准（如每条需求至少 1 条正向用例、关键交互有步骤与预期等）。

2. **登记任务**  
   在业务项目中使用 task_board：  
   - 若尚无任务卡文件，可先 `new-card` 再编辑内容，或直接编写任务卡文件；  
   - 使用 `add` 将任务加入 task-index，owner 为 **qa**，phase 为 **testing**，status 为 **todo**。

3. **明确提示用户**  
   向用户说明：  
   - 「请创建或打开【测试工程师】会话。」  
   - 「在该会话中运行 `python scripts/task_board.py list --role qa --status todo,doing` 找到本任务，使用 `claim` 领取后编写功能测试用例。」  
   - 「完成后使用 `update --status done --add-result <产物路径>` 更新任务；把产物路径或结论告诉我，我会回收并关闭任务。」

### 3.3 不执行的动作

- **不**在本会话中撰写测试用例文档。
- **不**替 qa 做「需求到用例」的分解与用例编写，即使注明「代做、越界」也不应默认采用。

### 3.4 回收与推进

- 当用户或 qa 会话反馈「测试用例已完成」并给出产物路径后，项目经理在 task_board 上将该任务更新为 done（若尚未更新）并记录 result_links。
- 根据出口条件判断是否进入下一阶段（如测试评审）；若采用任务板机制，以任务卡与索引为准。

---

## 4. 例外情形：「用户坚持在本会话内完成」

仅当用户**明确表示**不愿或暂时无法使用测试工程师会话、**坚持**在当前（项目经理）会话内完成测试用例时：

- 可注明：「由本会话代做测试用例产出，属越界，建议后续由测试工程师会话补正或确认。」
- 代做产出视为非正式/待补正，正式基线仍应以 qa 会话产出或 qa 补正版为准。

**重要**：用户仅说「开始你的工作吧」「帮我出测试用例」等，未明确拒绝使用 qa 会话时，**不得**主动采用该例外，而应走「派发任务 + 提示创建/使用 qa 会话」的流程。

---

## 5. 与其它阶段的一致性

上述逻辑同样适用于其他「归属明确为某一角色」的产出：

- **需求文档 / PRD / 用户故事** → 产品经理（product-requirements），PM 不代写。
- **评审结论与评审记录**（需求/设计/代码/测试）→ 对应评审角色（requirements-reviewer、design-reviewer、code-reviewer、test-reviewer），PM 不代写。
- **架构/设计文档** → 架构师、DBA 等，PM 不代写。

项目经理统一行为：**派发任务卡 → 提示对应角色会话领取并产出 → 回收产出并推进阶段**；仅在用户明确坚持且注明越界时，才在 PM 会话中代做并标注「建议由归属角色补正」。
