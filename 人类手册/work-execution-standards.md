# AI 工作执行标准

## 执行原则

- **分步验收**：每步有可观察验收点，通过后才能继续；不通过则分析原因、修复、重新验证
- **测试驱动**：先写可执行测试用例（如 pytest、Playwright 脚本），跑至失败后再实现业务代码至通过
- **完成标准**：标记完成前跑全量或相关测试套件，所有验收点与用例通过

## 执行环境

未特别说明时，开发与测试在本地环境执行（服务、pytest、E2E 等均在同一工作区所在机器运行）。

## 中止条件

遇到以下情况立即暂停：

1. 依赖无法自动安装（需要sudo、内网源、SSH密钥等）
2. 需要人工安装/配置（手动安装步骤、环境变量配置）
3. 需要人工确认信息（账号、网络可达性、软件安装状态）

暂停时列出：缺失项、安装/验证命令、待确认信息。待用户完成并确认后继续。

## 测试要求

- **冒烟用例覆盖**：启动服务、创建会话、事件可视化、远程可控、异常诊断
- **自给自足**：需要的数据通过实际操作创造（如通过网站操作触发验证码发送）
- **自生自灭**：测试产生的中间文件、数据、会话等必须在测试结束时删除
- **测试隔离**：用例间不依赖执行顺序；共享状态（限流、缓存等）在需时于用例内重置或通过单实例/单数据源避免互相影响
- **配置与生产一致**：E2E/集成测试中，凡生产代码从配置文件或环境读取的项（如认证账号、功能开关），测试应使用同一配置来源读取并用于断言或驱动，不得在测试中写死替代值；否则配置路径错误、加载失败等问题无法被测试发现，易出现「测试通过但线上/本机行为异常」

## 示例

**依赖问题**：
```
❌ 继续执行
✅ 暂停并列出：缺失项、安装命令、待确认信息
```

**验收点不通过**：
```
❌ 跳过进入下一步
✅ 分析原因 → 修复 → 重新验证 → 通过后继续
```
