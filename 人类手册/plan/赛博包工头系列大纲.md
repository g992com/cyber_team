# 《重生之我在AI世界当赛博包工头》系列故事大纲

## 定位与约定

- **题目**：重生之我在AI世界当赛博包工头  
- **主题**：通过软件工程、流程管理、CMMI 思想，建立并完善「用于软件开发的多智能体协作团队」的过程。  
- **风格**：简洁明快；用浅显话语讲专业理论；保持严谨（概念与规范库一致，可标注出处）。  
- **受众**：有一定编程基础的程序员或产品经理。  
- **素材来源**：以 [role-skills-design-memo.md](../role-skills-design-memo.md) 及规范库（.cyber_team/ 下的 process/、roles/、mapping/、skills/ 及人类手册/）为主线，必要时引用具体文件。当前采用单仓模式，规范根为 .cyber_team/；多根工作区已废弃，不推荐。

---

## 整体叙事弧

从「单人和一个 AI 对话写代码」到「你当包工头，指挥多角色 AI 按流程协作」。每一话解决一个「为什么/怎么做」的问题，并自然带出规范库中的对应设计（phases、roles、mapping、state、exit-criteria、多会话/蜂群等），最终落到「人读+机读」「单一事实来源」「门禁无例外」等原则。

**叙事定位**：故事**尚未完结**——你（作者/实践者）也在不断尝试，规范库与人类手册会持续演进。大纲中的「收尾与进化」是**阶段性小结**，不是终章；预留后续实践反馈、新变更备忘、新坑与对策，可随时增补新话或附录。

---

## 开篇

**第 0 话：重生在 AI 世界——你为什么是包工头而不是独行侠**

- 单 Agent 连续对话的局限：需求漂移、角色混用、难以复现。  
- 多角色协作的必然性：需求、架构、开发、测试、发布各司其职，和真实团队一样需要「流程+角色+验收」。  
- 点题：「包工头」不写每一块砖，但定工序、派活、验收、推进——你要建的是**规范与机制**，不是替 AI 写代码。  
- 可引用：role-skills-design-memo 中「从角色 skill 到团队活动规范」「人+Agent 双用」「流程为规范流程以约束 Agent 求稳定输出」。

---

## 第一部分：打地基——流程与阶段

**第 1 话：流程不是形式主义——为什么 AI 协作更需要「阶段」**

- 阶段化 = 可复现：需求→评审→设计→评审→开发→评审→测试→发布，每步有输入输出，Agent 才知道「当前该干什么、读什么、交什么」。  
- 规范库里的阶段全集：`.cyber_team/process/phases.yaml`（initiation → requirements → … → operations），id + order + outputs；不在这里堆细节，只建立「阶段是唯一事实来源」的直觉。  
- 金句方向：流程不是给领导看的，是给 Agent 和下一个角色看的。

**第 2 话：阶段全集与裁剪——CMMI/PMBOK 的「按上下文裁剪」怎么用在 Agent 身上**

- 规范库维护「阶段全集」，业务项目不复制阶段定义；用 `state.tailoring_snapshot` + `docs/process-tailoring.md` 表达「本项目执行哪些阶段、为什么」。  
- 裁剪谁来做：项目启动阶段由项目经理（或 project-initiation skill）根据用户需求做裁剪，输出项目计划与阶段清单。  
- 与 norm-improvement-plan 的「做法 B」一致：零拷贝、单一事实来源；Agent 读规范库 phases，按 tailoring_snapshot 过滤得到本项目阶段。

**第 3 话：门禁与出口条件——为什么「满足条件才能过」比「用户点一下就行」更稳**

- 出口条件可机读：`.cyber_team/process/exit-criteria.yaml`，每个 phase 的 required_artifacts、key_status（如 approved）；Agent/Orchestrator 据此判断能否推进。  
- 阶段转换规则：下一阶段由 `tailoring_snapshot` 顺序唯一确定；**出口条件未满足时不得推进**——要么继续补齐，要么中止并通报用户；不设「用户确认即可例外放行」。  
- 可引用：process.md 阶段转换规则、role-skills-design-memo 中「阶段推进无例外」相关备忘。

## 第二部分：招兵买马——角色与技能

**第 4 话：角色不是花名册——执行角色与评审角色为什么都要有**

- 执行角色：产出文档/代码（产品、架构、开发、测试、DevOps）；评审角色：审查与改进建议（需求评审、设计评审、代码评审、测试评审）。  
- 安全、DBA、SRE 等：设计评审专家兜底；代码层安全由代码评审在 PR 中覆盖（role-skills-design-memo 方案 A）。  
- 一句话：有人盖楼，有人验楼，分工清晰 Agent 才不「越权」或漏检。

**第 5 话：一角色多技能——产品经理为什么需要 PRD skill + 用户故事 skill**

- 从「一角色一 skill」到「一角色多 skill」：按文档类型/子领域拆，检索词与触发场景更清晰。  
- 产品经理：用户故事与验收标准、PRD/需求文档、可选故事地图/路线图；需求评审专家同理按 2～3 个 skill 拆。  
- 对应规范：.cyber_team/roles/、.cyber_team/mapping/、.cyber_team/skills/manifest.yaml 与各 SKILL.md；自建 skill 须遵循 create-skill 指引。

**第 6 话：阶段×角色×技能——mapping 表才是真正的「排班表」**

- `.cyber_team/mapping/phase-role-skill.yaml`：phase_id → role_id → skill_name；Agent 根据当前阶段与角色加载对应 skill。  
- 目标形态：由 Agent 决定并记录当前阶段与进展，自动启用对应角色与 skill，无需人工选 skill（role-skills-design-memo 技能触发从人工到 Agent 自决）。  
- 可顺带提：phases.yaml、roles.yaml、manifest 一起构成「配置中心」，多会话与蜂群都读这一套。

## 第三部分：工地规矩——规范与状态

**第 7 话：单一事实来源——phases、state、tailoring 为什么不能各写各的**

- 阶段定义唯一定义在 `.cyber_team/process/phases.yaml`；其他文档只引用不重复列举，避免多源不一致（role-skills-design 第 3 节改为引用 .cyber_team/process/phases.yaml）。  
- 业务项目：state 在业务项目根目录，含 current_phase、completed_phases、tailoring_snapshot；规范库只提供 `.cyber_team/process/state.yaml` 模板与 update_state.py，禁止把业务进展写回规范库。  
- 金句方向：一处定义，到处引用；状态只写业务项目，规范库只读不写。

**第 8 话：记忆在文件不在对话——会话丢了怎么接上**

- 可恢复的上下文都在持久化文件：state.yaml、task-index + 任务卡、project-docs-index、业务项目内的 .cyber_team/（process/、roles/、mapping/、skills/）。  
- 新会话恢复步骤：打开业务项目（单仓）、粘贴 pinned prompt、task_board list/claim、项目经理读 state + 任务分布。  
- 无法通过文件恢复的：仅存于对话的临时约定、多轮细微上下文；建议关键决策写进任务卡或阶段产出物（Cursor-多会话落地方案第七节）。

**第 9 话：任务卡与索引——让多会话别乱写**

- 任务卡（Markdown）+ 索引表（如 task-index.json）+ 脚本（task_board.py / update_state.py）：先查索引再读卡，更新通过脚本约束，减少格式漂移与误改。  
- project-docs-index.yaml：业务项目 docs 下，按阶段 id 列产出物路径；Agent 先读索引再按路径读文档，并遵守 artifact-metadata-convention（frontmatter）。  
- 谁初始化索引：由 PM/用户在项目启动或接入时复制模板并填写；依据为 phases 阶段 id 与实际/计划产出路径。

## 第四部分：开工——多会话与蜂群

**第 10 话：多开几个 tab 就是多智能体——Cursor 多会话落地方案入门**

- 不引入后端服务：用 Cursor 多会话（多 tab）模拟项目经理 + 产品 + 架构 + 开发 + 测试等；每个会话长期扮演一个角色，遵守规范库的流程与 skill 映射。  
- 统一 pinned prompt 骨架 + 各角色示例（项目总控、产品、架构、开发、测试、DevOps、各评审）；任务卡模板与多会话间流转。  
- 部署结构：单仓模式（业务项目内自带 .cyber_team 快照）；Rules 复制到业务项目 .cursor/rules/；Skills 从业务项目 .cyber_team/skills/ 读取；人类手册/Cursor-多会话协作落地方案 为操作依据。

**第 11 话：项目经理是触发式介入——什么时候 PM 出场、升级包怎么写**

- 不搞常驻调度：PM 在「跨角色冲突、阻塞升级、风险升级、计划偏离」时触发介入；谁发现谁触发，或用户问进展/持续 blocked 时 PM 主动接入。  
- 升级信息包（Escalation packet）：symptom/evidence/impact/options/decision_needed/recommended_next，减少口头争议。  
- PM 处置产出：问题分析与处置计划、更新后的阶段与活动清单、待澄清与待决策清单；role_updates、project_blockers、project_risks 进 state（README 进展回复契约）。

**第 12 话：从多会话到蜂群——Orchestrator + Role Agents 的配置中心长什么样**

- 同一套规范内核：.cyber_team/process/、.cyber_team/roles/、.cyber_team/mapping/、.cyber_team/skills/、state（业务项目）、project-docs-index；多会话是「人工版编排」，蜂群是「自动版编排」。  
- Orchestrator：读规范与 state，解析阶段与角色，下发 Task，收 Result，判断出口条件并推进阶段、更新 state。  
- Role Agents：按 role 加载 SOP 与 skill，读 project-docs-index 与 artifact 约定，产出并回传 Result。  
- 人类手册/多智能体蜂群编排落地方案 中的 Task/Result 数据模型、阶段驱动逻辑、分阶段落地路线可作为延伸阅读。

## 第五部分：进行中的收尾与进化（未完待续）

**第 13 话：出口条件未满足时怎么办——迭代、中止、推荐方案由谁提**

- 不得推进时的两条路：继续尝试满足条件，或判断无法达成则中止并通报用户。  
- 推荐方案由被阻塞阶段活动的负责人提出，经项目经理汇总后向用户通报，供用户选择（调整范围、回退补齐、记录为后续项等）。  
- 与「无用户确认例外」一致：门禁有约束力，可追溯性与流程一致性才成立（阶段推进无例外规则-已实施）。

**第 14 话：规范改进的「必要三项」——阶段唯一来源、出口条件可机读、产出物可发现**

- 三项必要改进（norm-improvement-plan）：阶段定义唯一事实来源（phases.yaml）；阶段出口条件可机读（exit-criteria.yaml）；产出物/模板可发现（project-docs-index + 先读索引再读文档）。  
- 其余有用项放入 norm-backlog，按需迭代；实施顺序在 norm-improvement-execution-plan 中已拆解（裁剪约定、唯一来源、state tailoring、出口条件、模板索引）。

**阶段性小结（非终章）：人读+机读、零拷贝、全工程回顾**

- 规范既要人读（流程说明、SOP、人类手册）也要机读（YAML、manifest、exit-criteria）；同一份约定，两种消费方式。  
- 零拷贝：业务项目不复制阶段定义，只存 tailoring_snapshot 与 process-tailoring 说明；模板与索引从规范库复制到业务项目时路径与责任明确。  
- 修改规范后的全工程回顾：id 一致性、mapping/manifest、人读/机读一致性（post-change-global-review 类规则），避免改一处漏多处。

**后续可写方向（随实践持续增补）**

- （历史/坑点示例）多根工作区下的真实坑：进展回复该引用哪个 README、state 写错仓库怎么办、路径悬空与引用修正；当前已改为单仓，规范根为 .cyber_team/。  
- 规范库自己的演进：人类手册目录重构、templates → project-docs 改名、规则集中到 .cyber_team/rules/、单仓与路径对齐——「规范也在长成」。  
- 蜂群落地与 demo：最小可行多 Agent 例子、orchestrator skill、根据多会话反馈迭代 pinned prompt。  
- 你踩过的坑与对策：以变更备忘为素材，每一条都可以变成一小节或一篇「包工头日记」。

## 写作与扩展建议

- 每话可控制在 2000～4000 字，便于单篇阅读与连载。  
- 专业术语首次出现时用一句话解释（如「出口条件」「tailoring_snapshot」），并可选标注「对应规范库 xxx」。  
- 可穿插「包工头日记」式小结或反例（如：没门禁时 AI 直接进开发、结果需求没冻结导致的返工），增强代入感。  
- **故事未完结**：你也在不断尝试，规范库与备忘会持续更新。新变更备忘、新坑与对策可随时增补为新话、新小节或「附：规范库更新说明」；上表「未用上或仅一笔带过的事件」可作为后续选题。
