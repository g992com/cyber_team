# PRD / 产品需求文档

## 文档结构

PRD 应包含以下部分，便于与用户故事衔接并被开发与测试执行：

| 部分 | 要点 |
|------|------|
| **背景与目标** | 问题/机会、业务目标、成功标准 |
| **用户与场景** | 目标用户、核心使用场景、关键用户旅程（便于与用户故事对应） |
| **范围** | 功能边界、在/不在范围内、与路线图对齐 |
| **优先级** | 必须/应该/可以；依赖与风险 |
| **验收标准** | 可验证条件、与用户故事可追溯 |
| **非功能与约束** | 性能/安全/合规要求；技术依赖、假设与限制条件（可合并进范围或单列） |
| **干系人对齐** | 关键决策与假设、已确认结论汇总；待澄清问题（独立小节+表格：序号、问题项、说明、影响范围）；可选澄清（独立小节+引导语「不影响可验收」+表格：序号、可选澄清问题、说明）（见下） |

编写时用 **【待确认】** 标出需用户决策的项，**已确认** 标出已明确的结论；可维护「已确认结论汇总」便于查阅。文档路径建议：`docs/PRD-{项目或功能名}.md`。

### 干系人对齐下的「待澄清问题」与「可选澄清」

- **待澄清问题**：所有**必须澄清**的问题放在「干系人对齐」下的「待澄清问题」小节，用表格记录，便于跟踪与评审。表格列建议：**序号**、**问题项**、**说明**、**影响范围**。模板示例：

```markdown
| 序号 | 问题项 | 说明 | 影响范围 |
|------|--------|------|----------|
| 1    | （待澄清的问题描述） | （背景或选项） | （功能/模块/验收） |
```

- **可选澄清**：仅影响实现或配置、可由实现方自定的模糊点，不放入「待澄清问题」表；在「干系人对齐」下单独设「可选澄清」小节（可编号，如 7.2），用表格列出，便于开发前按需与产品/配置确认。小节开头建议用一句引导语：**以下问题不影响「可验收」判定，开发前可按需与产品/配置确认，以减少实现歧义：**。表格列建议：**序号**、**可选澄清问题**、**说明**。设计稿中的专有名词（如某「形象」、某「状态」）若含义可由上下文推断，在「说明」列或「假设与限制」中简要说明即可。

**可选澄清表格模板**：

```markdown
| 序号 | 可选澄清问题 | 说明 |
|------|--------------|------|
| 1    | （问题简述，如：某配置来源与可选枚举？） | （背景或可约定方，如：可由产品/配置单独约定。） |
```

## 编写与澄清流程

1. **明确输入**：现有需求、用户故事、路线图或迭代范围；若有可检视的上下文（DB、代码、配置），先读取以 grounding 后续提问。
2. **按领域分组澄清**（见下表）：一次提问聚焦 1～2 个领域，逐批澄清；缺失项标注「待定」并列出依赖。
3. **填充结构**：按文档结构逐项填写；根据用户回复将【待确认】改为已确认并写入文档。
4. **收到澄清后的更新**：同步做四件事——（1）正文中该处改为「已确认」并写清结论；（2）「已确认结论汇总」中补充该条；（3）从「待澄清问题」小节表格中删除该行；（4）变更记录增加一条（日期、澄清更新、简要说明）。
5. **验收标准**：每条可对应到可测试条件或用户故事验收标准。
6. **对齐与更新**：与干系人确认范围与优先级；变更时更新 PRD 并维护追溯（见「变更与追溯」）。
7. **范围边界**：若某类需求（如数据表结构、字段定义）由后端/数据侧单独定义，在范围或非功能约束中写明「本 PRD 不做 XX 表/字段级需求」，避免 PRD 与实现职责混淆。
8. **迭代直至定稿**：每轮更新后检查是否还有需澄清的点；区分**必须澄清**（影响实现）与**可选澄清**（可由实现方自定）。
9. **发布前评审**：产品 + 研发 + 测试（及可选设计/业务）对 PRD 做一次性评审；输出「通过 / 有条件通过 / 退回」，结论记入「干系人对齐」或单独「评审结论」，与需求评审阶段衔接。
10. **终稿检查**：核心功能、非功能、安全、接口均已写明；与既有用户故事或 Backlog 无冲突；待澄清问题已在干系人对齐中单独小节+表格列出。
11. **是否拆故事**（可选）：定稿后、进入设计/开发前，按「是否需要拆分为用户故事」判断标准执行并输出结论；若结论为建议拆故事，可再调用 user-story 类能力进行拆分。

### 按领域分组澄清（常见澄清点）

| 领域 | 常见澄清点 |
|------|------------|
| **核心逻辑** | 匹配方式（精确/模糊/大小写）、多条数据时的返回规则、无结果时的格式 |
| **输入与校验** | 空值/长度/特殊字符、前后端校验 |
| **访问形态** | 仅 API / Web 页 / 两者、批量 vs 单次 |
| **安全与限流** | 限流维度（IP/用户）、阈值、超限表现、访问控制 |
| **性能** | 预期 QPS/PV、缓存策略、部署形态 |
| **日志与审计** | 存储方式、格式、去重规则、访问入口、访问控制 |
| **异常** | 异常时页面提示、错误码约定 |

## 与用户故事的衔接

- 将 PRD 范围拆为可交付的用户故事；每个故事具备角色、目标、价值、验收标准（INVEST）。
- PRD 的验收标准可映射到故事级验收标准，便于测试覆盖。

### 是否需要拆分为用户故事（智能体可执行判断标准）

**执行时机**：PRD 定稿或发布前评审后，进入开发/设计前。智能体读取当前 PRD 与上下文（是否存在 Backlog/迭代计划、团队约定等），按下列维度检查并输出结论。

**判断维度**（每项为「是」则计 1 分，倾向拆故事）：

| 维度 | 可检查条件 | 满足则计 1 分 |
|------|------------|----------------|
| **范围规模** | PRD 中独立功能点或验收标准条数 > 5，或明确涉及 ≥2 个模块/子系统 | 是 |
| **多角色/多场景** | PRD「用户与场景」中明确 ≥2 类不同角色，或 ≥3 个独立使用场景 | 是 |
| **分迭代/Backlog** | 存在或计划按迭代/Sprint/Backlog 交付，或需按「任务包」分配给多智能体/多人 | 是 |
| **故事级追溯** | 团队或流程要求：测试用例/评审结论需追溯到「单条可交付单元」（故事级） | 是 |
| **验收标准未按可交付拆** | PRD 验收标准多为「整块功能」级，单条无法对应一次实现+测试的闭合单元 | 是 |

**判定规则**：

- **合计 ≥2 分** → 结论：**建议拆分为用户故事**；输出依据（满足的维度列表），并建议按 INVEST 将 PRD 拆为故事后再进入开发。
- **合计 0～1 分** → 结论：**可直接按 PRD 开发**；输出依据，并注明「以 PRD 验收标准为可执行与测试追溯单位」。

**输出格式**（智能体必须输出，便于留痕与人工复核）：

```text
[是否需要 User Story]
结论：建议拆分为用户故事 / 可直接按 PRD 开发
依据：维度1(是/否)、维度2(是/否)、…；合计 N 分。
补充：若有团队显式约定（如「一律拆故事」或「一律不拆」），以约定为准并在此注明。
```

## 原则

1. **不越界**：仅澄清需求，不替用户做产品决策；选项不清时提供建议，由用户拍板。
2. **可落地**：每个已确认项应对应可验收的交付物。
3. **适度抽象**：能由实现方合理推断的细节，可不写入 PRD。
4. **回应为先**：收到回复后优先更新文档，再发起下一轮提问。
5. **单源真相**：PRD 为该需求范围的权威描述；设计/开发/测试以 PRD 为准，变更通过更新 PRD 并留痕（见「变更与追溯」）。

## 变更与追溯

- **范围或优先级变更**：更新 PRD 并注明日期与原因；结构性变更建议再走一轮发布前评审。
- **小措辞修正**：可直接修改，无需评审。
- 在 PRD 文末或团队约定处维护变更记录（日期、变更类型、简要说明），便于与设计/开发/测试追溯。

## 提问示例（澄清时）

**领域式**：关于查询逻辑：关键字是精确匹配还是模糊？同一关键字多条记录时，返回最新一条还是全部？

**选项式**：限流超限时：返回 HTTP 429，还是在页面上提示「请求过于频繁」即可？

**补充式**：无数据关键字需要单独记录。去重周期是：按两次落盘之间，还是同一关键字全局永久只记一次？

## 输出验收

- 范围与优先级明确；验收标准可验证、可追溯。
- 与既有用户故事或 Backlog 无冲突；待澄清问题已完成澄清并更新记录。
- 若存在仅影响实现细节的模糊点，列入「可选澄清」，并经用户确认是否补充。
